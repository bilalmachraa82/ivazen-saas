Área 1 — Segurança Social
P1

Agostinho — SS só mostra vendas de Janeiro

"No ecrã Segurança Social - Declaração, apenas as vendas de Janeiro são apresentadas. Deviam aparecer as vendas de todo o trimestre."
✓ Resolvido
Causa

O endpoint sync-efatura retornava erro da AT (AT_SCHEMA_RESPONSE_ERROR) para este cliente, impedindo a importação de vendas dos meses seguintes. Sem fallback, o sistema ficava apenas com os dados de Janeiro.

Correcção implementada
  • Implementado fallback chain em sync-efatura: quando a API de vendas da AT falha, o sistema tenta automaticamente via sync-recibos-verdes
  • A lógica de agregação trimestral em useSocialSecurity já estava correcta — o problema era falta de dados
  • Após re-sync com o fallback activo, os 3 meses do trimestre ficam disponíveis
Nota AT

Este cliente tem um erro intermitente na API da AT (AT_SCHEMA_RESPONSE_ERROR). O fallback chain mitiga o problema automaticamente, mas a causa raiz é upstream (AT).

P2

Cátia — SS mostra 0,00€ e valores incorrectos

"O ecrã de Segurança Social mostra o valor a zeros. O valor total acumulado dos trimestres não coincide com os valores da Segurança Social."
✓ Resolvido
Causa

Mesma causa raiz que P1 — falha na importação de vendas impedia o cálculo SS. Adicionalmente, a base de cálculo SS usa total_amount - total_vat (base tributável, sem IVA) enquanto os totais visíveis na UI mostravam o valor com IVA.

Correcção implementada
  • Fallback chain garante importação completa de vendas
  • Cálculo SS usa correctamente a base tributável (sem IVA) por categoria CAE/CIRS
  • Os coeficientes SS estão definidos em ssCoefficients.ts por tipo de actividade
P3

Gestão de isenção SS — datas de enquadramento

"No caso de um cliente isento até 31/11/2024, o sistema deve considerar o fim da isenção e processá-lo como contribuinte a partir de 01/12/2024."
✓ Resolvido
Causa

A lógica de isenção verificava apenas o campo is_first_year sem considerar a data de início de actividade para calcular quando a isenção termina.

Correcção implementada
  • O perfil regista agora ss_start_date — a data exacta em que o contribuinte passa a ser obrigado a contribuir
  • SocialSecurity.tsx verifica is_first_year e ss_start_date do perfil
  • Clientes cuja isenção já terminou são automaticamente processados como contribuintes SS activos
P4

Regras de actividade SS — valores sem IVA e padrão −25%

"Valores relativos a subsídios à exploração, produção de energia e alojamento local devem ser preenchidos sem IVA. O sistema deve adoptar como padrão uma variação de −25%."
✓ Resolvido
Causa

A base de cálculo SS não separava correctamente por código de actividade (CAE/CIRS). O ajuste de −25% (variação padrão para a declaração trimestral) não estava implementado como default.

Correcção implementada
  • Coeficientes SS definidos por categoria em ssCoefficients.ts
  • Base de cálculo usa total_amount − total_vat (sem IVA) para todas as categorias
  • A variação de −25% é aplicada como padrão na declaração trimestral SS
  • Actividades específicas (subsídios, energia, alojamento local) mapeadas aos respectivos códigos CIRS
Área 2 — Processamento de Facturas
P5

Erro 500 ao carregar facturas (Edge Function falha)

"Ocorrem falhas ao carregar faturas (Edge Function com status non-2xx). As faturas de vendas, mesmo em PDF, não estão a ser processadas."
✓ Resolvido
Causa

A variável AI_API_KEY (Google Gemini) não estava configurada nos Supabase Secrets de produção. Todas as 7 Edge Functions que processam PDFs falhavam com erro de autenticação ao chamar a API Gemini.

Correcção implementada
  • AI_API_KEY sincronizada nos Supabase Secrets de produção
  • 7 Edge Functions restauradas: extract-invoice-data, extract-withholding, process-email, extract-recibo-verde, sync-efatura, sync-recibos-verdes, extract-modelo10
  • PDFs de vendas e compras voltam a ser processados correctamente
Área 3 — Dashboard
P6

Rafael Paisano — Dashboard mostra "IVA Trimestral" (regime é Mensal)

"No cliente Rafael Paisano, o sistema apresenta um aviso de IVA Trimestral quando o regime correcto é IVA Mensal."
✓ Resolvido
Causa

Quando o campo vat_regime era null na base de dados, o Dashboard usava quarterly como fallback. Rafael tem regime mensal mas o campo não tinha sido preenchido.

Correcção implementada
  • Dashboard.tsx (L88-93): quando vat_regime é null, o default é agora monthly (prazo dia 20 do mês seguinte)
  • Regimes isentos (exempt_53, exempt_9) continuam correctamente em quarterly
  • O alerta mostra agora o prazo correcto conforme o regime configurado
Referência — Ronda 1, P1

Este problema está relacionado com o P1 da Ronda 1. A correcção da Ronda 1 introduziu o formatVatRegime() e a migração de 432 perfis. A Ronda 2 corrigiu o fallback de null (que a migração pode não ter coberto para todos os perfis).

Área 4 — Perfil do Cliente
P7

Perfis mostram periodicidade IVA igual para todos os clientes

"As informações no perfil aparecem iguais para todos os clientes. O sistema deve recolher individualmente esta informação no portal da AT."
⚠ Feature futura
Diagnóstico

Actualmente, o regime IVA (vat_regime) é configurado manualmente no perfil do cliente. Cada contabilista pode definir individualmente: Normal mensal, Normal trimestral, Isento Art. 53º ou Isento Art. 9º. O sistema não busca esta informação automaticamente ao portal da AT.

O que já funciona
  • O perfil tem selector com as 4 opções de regime IVA
  • Ao definir o regime, o Dashboard, SS e Apuramento usam o valor correcto
  • A migração da Ronda 1 (432 perfis) actualizou os regimes mais comuns
Feature planeada

A detecção automática do regime IVA requer scraping do portal da AT (Cadastro Contribuinte), que não está disponível via API. Esta funcionalidade está planeada como melhoria futura. Até lá, o regime deve ser configurado manualmente por cliente nas Definições > Regime IVA.

Filtros da tabela de Compras não funcionam

Área 6 — Vendas e Apuramento
P10

Vendas deixaram de aparecer após última correcção

"Após a última correção, as faturas de vendas deixaram de aparecer por completo. Praticamente nada foi corrigido nestas duas secções."
✓ Resolvido
Causa

Duas causas combinadas: (1) a AI_API_KEY em falta impedia o processamento de PDFs de vendas; (2) o sync-efatura não tinha fallback quando a API da AT retornava erro ou lista vazia para vendas.

Correcção implementada
  • AI_API_KEY restaurada (ver P5) — PDFs voltam a ser processados
  • sync-efatura refactored com resolveAccountantConfig() para gerir credenciais do contabilista de forma robusta
  • Fallback chain: quando a query de vendas AT retorna vazio ou erro, o sistema chama automaticamente sync-recibos-verdes
  • sync-recibos-verdes configurado com verify_jwt=false em config.toml para aceitar chamadas internas do fallback
Validação técnica

Edge Functions validadas com deno check. 14 testes E2E (Playwright) passam. 867 testes unitários passam. Build de produção sem erros.
